# PRD: Sync Skill — Section Text File Support
## Status: Complete
## Last Updated: 2026-03-10

---

## 1. Overview

The `/sync-innovation-images` skill currently reads section images from numbered source folders (e.g. `01 BIM MEP Engineering/`) and updates only image fields in `section-description.json`. A new file format is being introduced: each source section folder now contains a `Section - {Name}.txt` file alongside the numbered images (`01.png`, `02.png`, etc.). This text file is the single source of truth for the section's textual content (tagline, description, bullet capabilities) and declares which images are public (`ON`) or confidential (`OFF`). The skill must be updated to parse these text files, update the relevant JSON fields, set `confidential` flags on images, and copy the numerically-named image files using zero-padded numeric filenames.

---

## 2. Goals

- **G1** — The skill parses `Section - {Name}.txt` files and maps their content to the correct fields in `section-description.json`.
- **G2** — Image entries in the text file marked `ON` produce `confidential: false` (or omit the flag); entries marked `OFF` produce `confidential: true` in the JSON images array.
- **G3** — Incoming image files named `01.png`, `02.png`, etc. are copied to the data folder as `001.png`, `002.png`, etc. (three-digit zero-padded, no slug appended).
- **G4** — The skill leaves untouched JSON fields (`id`, `slug`, `order`, `title`, `stats`, `links`, `video`) exactly as they are.
- **G5** — The skill runs end-to-end without manual edits when the source folder contains the text file and numbered images.

---

## 3. User Stories

- **US-1** — As a content manager, I drop a `Section - Green Finance.txt` and `01.png`…`06.png` into the source folder, run `/sync-innovation-images`, and the JSON and public assets update automatically.
- **US-2** — As a content manager, I mark an image line `OFF` in the text file to flag it confidential, and the JSON `confidential: true` is written without touching other data.
- **US-3** — As a developer, I run the sync skill and expect existing fields (`id`, `slug`, `stats`, `links`) to be preserved unchanged.
- **US-4** — As a content manager, I don't need to rename incoming images — files named `01.png` are enough; the skill handles renaming to `001.png`.

---

## 4. Functional Requirements

### Text file parsing

- **FR-1** — The skill looks for a file matching `Section - *.txt` (case-insensitive) inside each source section folder. If found, it parses it. If not found, image-only sync proceeds as before (backward-compatible).
- **FR-2** — The `Tagline:` line maps to `shortDescription`. Everything after `Tagline:` on the same line (trimmed) is the value.
- **FR-3** — The `Description:` block maps to `longDescription`. It spans from the line after `Description:` to the line before `Bullets:`. Paragraphs are joined with `\n\n`. The literal string `sect` appearing anywhere in the description block is treated as a typo and stripped (replaced with an empty string, then whitespace normalised).
- **FR-4** — The subItems block header may appear as any of the following (case-insensitive, colon optional): `Bullets`, `Capabilities`, `Services`, `Key Services`. The skill detects whichever variant is present and treats the `- ` prefixed lines that follow as `subItems`. Each `- ` prefixed line becomes one `{ "label": "...", "slug": "..." }` entry. The slug is generated by lowercasing the label, replacing spaces with hyphens, removing special characters (`&`, `(`, `)`, `/`, `—`), and collapsing multiple hyphens.
- **FR-5** — The `Images:` block maps to the image entries. Each line has the format: `{ON|OFF} - {number} - {caption text}`. Lines that do not match this pattern are skipped.
- **FR-6** — `ON` → `confidential` field omitted (or set to `false`). `OFF` → `confidential: true`.
- **FR-7** — The number from each image line (e.g. `01`, `02`) is used to find the corresponding image file in the source folder (`01.png`, `01.jpg`, `01.jpeg`).
- **FR-8** — The caption text (everything after `{number} - `) is written as both `alt` and `caption` in the JSON image entry. HTML entities in the caption (e.g. `&amp;`) are decoded to their plain-text equivalents (`&`). Stray HTML artifacts (e.g. `divdiv`) are stripped.

### Image file handling

- **FR-9** — Incoming image files are named with two-digit numbers (`01.png`, `02.png`). The skill copies them to the data folder as three-digit zero-padded names: `001.png`, `002.png`, `003.png`.
- **FR-10** — Before copying, delete any existing `001.*`, `002.*`, `003.*`, etc. files in the destination data folder (old gallery images).
- **FR-11** — The `src` field in the JSON image entry is `"./001.png"`, `"./002.png"`, etc.
- **FR-12** — There is no "main image" concept in the new format. No `main-bg.*` is written. The component already uses `images[0]` as the card/modal hero image.
- **FR-13** — Images whose source file does not exist on disk (referenced in the text file but no matching `{number}.png/jpg/jpeg` found) are skipped with a warning.

### JSON update

- **FR-14** — The skill reads the existing `section-description.json`, merges only the parsed fields (`shortDescription`, `longDescription`, `subItems`, `images`), and writes the file back. All other fields remain unchanged.
- **FR-15** — The `images` array in JSON is fully replaced by the entries derived from the text file's `Images:` block (respecting `ON`/`OFF`).
- **FR-16** — The `title` field is NOT overwritten. It stays as the existing value in the JSON.

### Mapping

- **FR-17** — Section folder-to-data-folder mapping continues to use the same fuzzy keyword matching as before (BIM, Climate, Daylight, Fluid/CFD, Finance, Circular). The text file name (`Section - Green Finance.txt`) can also be used as a secondary mapping hint: slugify the name part and match against data folder names.

### Backward compatibility

- **FR-18** — If a source section folder has no `Section - {Name}.txt`, the skill processes images only (as before), without touching `shortDescription`, `longDescription`, or `subItems`.
- **FR-19** — If a source folder has a text file but no image files, the `images` array is set to `[]` and text fields are still updated.

---

## 5. Non-Goals / Out of Scope

- No changes to `title`, `id`, `slug`, `order`, `stats`, `links`, or `video` fields.
- No UI for editing the text files — authors edit `.txt` files directly in the filesystem.
- No validation of the text file beyond what is needed to parse it.
- No support for animated GIFs as main images (`animated` field stays `false` for all entries in this flow).
- No changes to the asset pipeline script (`copy-service-assets.sh`) itself.
- No changes to the React components that consume the JSON data.

## 7. Technical Considerations

- **File:** `.claude/commands/sync-innovation-images-ces-portfolio.md` — the entire skill is a markdown prompt with embedded bash examples. The update is purely a documentation/instruction change to the skill file; Claude executes the steps when invoked.
- **Text parsing:** The skill instructs Claude to read the `.txt` file, split on section headers (`Tagline:`, `Description:`, `Bullets:`, `Images:`), and process each block.
- **HTML entity decoding:** `&amp;` → `&`. Stray `divdiv` or similar HTML artifacts are stripped with a simple regex or string replacement.
- **Slug generation for subItems:** Lowercase → replace spaces with `-` → remove `& ( ) / — ,` → collapse `--` → trim `-`.
- **Image filename collisions:** If both `01.png` and `01.jpg` exist, prefer `.png`. Warn if multiple formats found.
- **No new npm packages or env vars needed** — this is a markdown skill file update only.
